# 校园二手交易平台 — 技术栈说明与功能介绍

> 本文档适用于 PBL（Project-Based Learning）项目报告，系统梳理「校园二手」平台的技术架构与各模块功能设计。

---

## 一、项目背景与目标

### 1.1 项目背景

高校师生在日常学习生活中会产生大量闲置物品，如教材、电子产品、生活用品等。传统线下交易渠道信息分散、效率低；通用电商平台又缺乏校园社区的信任感与就近交易优势。本项目旨在构建一个**面向校园社区的二手交易平台**，让闲置物品在校内高效流通。

### 1.2 项目目标

| 目标维度 | 具体目标 |
|----------|----------|
| 业务目标 | 实现商品发布、浏览、购物车、下单、订单履约、即时聊天等完整交易闭环 |
| 技术目标 | 采用前后端分离架构，前端 SPA + 后端 RESTful API，支持实时消息推送 |
| 体验目标 | 移动端优先，界面简洁可信，符合校园社区使用场景 |
| 扩展目标 | 预留 AI 购物助手接口，支持智能推荐与问答 |

### 1.3 系统定位

- **产品名称**：校园二手
- **目标用户**：在校学生、教职工等校园社区成员
- **核心价值**：就近交易、社区信任、操作便捷

## 二、系统总体架构

### 2.1 架构模式

本系统采用**前后端分离（B/S 架构）**：

```
┌─────────────────────────────────────────────────────────┐
│                      浏览器客户端                          │
│  Vue 3 SPA（端口 5173）                                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ 页面视图  │ │ 状态管理  │ │ 路由守卫  │ │ API 封装  │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │
└────────────────────────┬────────────────────────────────┘
                         │ HTTP REST / WebSocket
                         ▼
┌─────────────────────────────────────────────────────────┐
│                   Express 后端（端口 3000）                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ 路由/API  │ │ 业务逻辑  │ │ 文件上传  │ │ Socket.IO │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │
└────────────────────────┬────────────────────────────────┘
                         │ SQL
                         ▼
┌─────────────────────────────────────────────────────────┐
│              MySQL 数据库（2_shou）                        │
│  users / goods / cart_items / orders / order_items /     │
│  messages / addresses                                    │
└─────────────────────────────────────────────────────────┘
```

### 2.2 部署与运行

| 组件 | 地址 | 说明 |
|------|------|------|
| 前端开发服务 | `http://localhost:5173` | Vite Dev Server，开发时代理 API |
| 后端服务 | `http://localhost:3000` | Express + Socket.IO 同端口 |
| 数据库 | `localhost:3306` | MySQL 8.x，库名 `2_shou` |
| 静态资源 | `/uploads/*` | 用户上传的商品图片 |

开发环境下，Vite 将 `/api` 与 `/uploads` 请求代理至后端，避免跨域问题。

## 三、技术栈详细说明

### 3.1 前端技术栈

#### 3.1.1 核心框架：Vue 3

| 项目 | 说明 |
|------|------|
| 版本 | Vue 3.5.x |
| 选型理由 | Composition API 逻辑复用清晰；响应式性能优于 Vue 2；生态成熟 |
| 应用方式 | 单文件组件（SFC，`.vue`），`<script setup>` 语法 |

Vue 3 负责整个用户界面的渲染与交互。页面按功能拆分为独立视图组件（如 `HomeView`、`CartView`），公共 UI 抽取为可复用组件（如 `SafeImage`、`FetchErrorAlert`）。

#### 3.1.2 构建工具：Vite 8

| 项目 | 说明 |
|------|------|
| 版本 | Vite 8.x |
| 插件 | `@vitejs/plugin-vue` |
| 特性 | 基于 ESM 的极速热更新（HMR）；生产构建由 Rollup 打包 |

Vite 作为现代前端构建工具，开发阶段无需完整打包即可启动，显著缩短冷启动与热更新时间。`vite.config.ts` 中配置了开发代理，将 API 请求转发至后端。

#### 3.1.3 类型系统：TypeScript

| 项目 | 说明 |
|------|------|
| 版本 | TypeScript 5.9.x |
| 工具 | `vue-tsc` 做类型检查 |
| 收益 | 接口类型约束、IDE 智能提示、编译期错误发现 |

前端 API 层、Store、工具函数均使用 TypeScript 编写，提升代码可维护性与团队协作效率。

#### 3.1.4 路由管理：Vue Router 4

| 项目 | 说明 |
|------|------|
| 模式 | `createWebHistory`（HTML5 History） |
| 核心能力 | 嵌套路由、懒加载、路由元信息（meta）、全局守卫 |

路由设计要点：

- **登录页** `/login`：标记 `guest: true`，已登录用户自动跳转首页
- **主布局** `/`：嵌套所有业务页面，标记 `requiresAuth: true`
- **路由守卫**：未登录访问受保护页面时，重定向至登录页并携带 `redirect` 参数，登录成功后回跳

#### 3.1.5 状态管理：Pinia 3

| 项目 | 说明 |
|------|------|
| Store | `user`（用户登录态）、`cart`（购物车数据） |
| 持久化 | 用户信息存于 `localStorage`，刷新页面保持登录 |

Pinia 替代 Vuex 作为官方推荐状态管理方案，API 更简洁。用户 Store 管理登录/登出/初始化；购物车 Store 封装列表拉取与加购逻辑，供多页面共享。

#### 3.1.6 UI 组件库：Ant Design Vue 4

| 项目 | 说明 |
|------|------|
| 版本 | Ant Design Vue 4.2.x |
| 图标 | `@ant-design/icons-vue` |
| 定制 | `a-config-provider` 注入主题 Token |

选用 Ant Design Vue 提供表单、表格、卡片、Modal、Tabs 等成熟组件，缩短开发周期。主题色定制为 **Warm Campus Teal**（`#0f857b`），区别于默认蓝色，体现校园品牌识别。

#### 3.1.7 HTTP 客户端：Axios

| 项目 | 说明 |
|------|------|
| 版本 | Axios 1.13.x |
| 封装 | 统一 `baseURL`、`timeout`（15s）、响应拦截 |
| API 模块 | `frontend/src/api/index.ts` 集中定义所有接口 |

所有前后端数据交互通过 Axios 发起 REST 请求。文件上传使用 `FormData` 单独调用 upload 接口。

#### 3.1.8 实时通信：Socket.IO Client

| 项目 | 说明 |
|------|------|
| 版本 | socket.io-client 4.8.x |
| 用途 | 聊天消息实时推送 |
| 机制 | 用户登录后 `emit('join', userId)` 加入个人房间，收到 `new_message` 事件后刷新消息列表 |

#### 3.1.9 样式方案

| 层级 | 方案 |
|------|------|
| 组件样式 | Ant Design Vue 内置样式 + `dist/reset.css` |
| 全局设计令牌 | `style.css` 中 CSS 变量（OKLCH 色彩、间距、圆角） |
| 响应式 | 媒体查询 + Ant Design 栅格（`:xs` `:sm` `:md` 等） |
| 移动端适配 | 底部导航、Safe Area、最小触控区域 44px |

### 3.2 后端技术栈

#### 3.2.1 运行环境：Node.js

后端基于 Node.js 运行时，使用 CommonJS 模块规范（`require`/`module.exports`），便于快速搭建 REST API 服务。

#### 3.2.2 Web 框架：Express 5

| 项目 | 说明 |
|------|------|
| 版本 | Express 5.2.x |
| 中间件 | `cors`（跨域）、`express.json`（JSON 解析，上限 10MB） |
| 职责 | 路由分发、请求处理、静态文件服务 |

Express 作为轻量级 Web 框架，负责注册全部 `/api/*` 路由，并将 `/uploads` 目录映射为静态资源访问路径。

#### 3.2.3 数据库驱动：mysql2

| 项目 | 说明 |
|------|------|
| 版本 | mysql2 3.20.x |
| 连接方式 | 连接池（`createPool`），最大 10 连接 |
| 特性 | 支持 Promise/async-await；参数化查询防 SQL 注入 |

数据库操作全部采用参数化 SQL，避免字符串拼接带来的注入风险。

#### 3.2.4 文件上传：Multer

| 项目 | 说明 |
|------|------|
| 版本 | Multer 2.2.x |
| 存储 | 磁盘存储（`uploads/` 目录） |
| 限制 | 单文件最大 5MB；多图最多 9 张 |
| 命名 | 时间戳 + 随机字符串，避免文件名冲突 |

#### 3.2.5 实时通信：Socket.IO Server

| 项目 | 说明 |
|------|------|
| 版本 | socket.io 4.8.x |
| 挂载 | 与 Express 共用 `http.Server` |
| CORS | 允许 `localhost:5173`、`localhost:4173` |

消息发送成功后，服务端向接收者所在房间推送 `new_message` 事件，实现聊天实时更新。

#### 3.2.6 跨域：CORS

开发阶段前后端分端口运行，通过 `cors` 中间件允许浏览器跨域访问 API。

### 3.3 数据库：MySQL 8

#### 3.3.1 基本信息

| 项目 | 说明 |
|------|------|
| 版本 | MySQL 8.0.x |
| 字符集 | utf8mb4（支持中文及 Emoji） |

#### 3.3.2 数据表设计

| 字段 | 类型 | 说明 |

**goods — 商品表**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int, PK, 自增 | 商品 ID |

| pay_time/ship_time/receive_time | datetime | 各节点时间戳 |

**order_items — 订单明细表**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int, PK | 明细 ID |
| order_id | int, 索引 | 所属订单 |
| goods_id | int | 商品 ID |
| goods_content | text | 商品描述快照 |
| goods_price | decimal(10,2) | 成交单价快照 |

**messages — 消息表**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int, PK | 消息 ID |
| sender_id | int, 索引 | 发送者 |

后端启动时自动读取 `migrate.sql` 执行建表语句（`CREATE TABLE IF NOT EXISTS`），保证新环境一键初始化。

## 四、功能模块详细介绍

### 4.1 用户认证模块

#### 4.1.1 功能描述

提供用户注册、登录、退出及登录态保持能力，是所有业务功能的前置条件。

#### 4.1.2 功能细节

**注册**

- 独立注册页 `/register`，与登录页风格一致（Warm Campus Teal 卡片）
- 字段：用户名（3-32 位）、密码（≥6 位）、确认密码、用户协议勾选
- 密码强度可视化指示；用户名格式与唯一性后端校验
- 注册成功后自动登录并跳转首页

**登录**

- 校验用户名与密码，成功返回用户对象（含 id、username、avatar、role、status）
- 禁用账号（status=0）拒绝登录
- 前端将用户信息写入 Pinia Store 及 `localStorage`
- 支持登录后跳转至登录前访问的页面（`redirect` 参数）

**路由守卫**

- 未登录访问 `/cart`、`/publish` 等页面 → 强制跳转 `/login`
- 已登录访问 `/login` → 自动跳转首页

**退出**

- 清除 Store 与 localStorage，跳转登录页

#### 4.1.3 业务流程

```
访客 → 注册/登录 → 获取用户信息 → 进入首页
                ↓
         localStorage 持久化
                ↓
         刷新页面自动恢复登录态
```

### 4.2 商品浏览与搜索模块

#### 4.2.1 功能描述

首页展示平台全部商品，支持关键词搜索，用户可点击进入详情页查看完整信息。

#### 4.2.2 功能细节

**商品列表**

- 调用 `GET /api/goods`，默认按 ID 倒序（最新在前）
- 卡片展示：封面图、描述、价格、卖家昵称
- 已售商品（status=2）显示「已售」标签
- 无图商品显示占位图标

**关键词搜索**

- 搜索框输入关键词，按商品 `content` 字段模糊匹配（`LIKE %keyword%`）
- 支持回车或点击搜索按钮触发

**商品详情**

- 路由 `/goods/:id`，展示多图轮播、价格、卖家、在售/已售状态
- 无效 ID 或已删除商品显示错误提示与重试

**空状态与异常**

- 无商品：引导用户发布
- API 失败：显示错误条，支持一键重试

#### 4.2.3 数据流

```
HomeView → api.getGoods() → Express → MySQL goods JOIN users
    ↓
渲染商品卡片 → 点击 → GoodsDetailView → api.getGoodsDetail(id)
```

### 4.3 商品发布模块

#### 4.3.1 功能描述

登录用户可发布二手商品，填写描述与价格，上传最多 9 张图片。

#### 4.3.2 功能细节

**发布表单**

| 字段 | 校验规则 |
|------|----------|
| 商品描述 | 必填，≤500 字 |
| 价格 | 必填，>0 |
| 图片 | 可选，最多 9 张，单张 ≤5MB |

**图片上传流程**

1. 用户选择本地图片
2. 前端调用 `POST /api/upload`，Multer 存至 `uploads/`
3. 返回 `/uploads/xxx.jpg` 相对路径
4. 发布时将路径数组 JSON 化写入 `goods.images`

**发布提交**

- 调用 `POST /api/goods`，默认 status=1（在售）
- 成功后跳转首页，新商品立即可见

**卖家管理**

- 商品详情页，发布者可见「标记已售」按钮
- 调用 `PUT /api/goods/:id/status`，将 status 设为 2

### 4.4 购物车模块

#### 4.4.1 功能描述

买家可将感兴趣的商品加入购物车，统一管理待购商品，支持批量选择与结算。

#### 4.4.2 功能细节

**加入购物车**

- 商品详情页点击「加入购物车」
- 后端校验：商品必须在售；不能购买自己的商品
- 同一商品重复加购：数量 +1（非新增条目）

**购物车页面**

- 展示商品缩略图、描述、单价、数量、卖家
- 支持修改数量（InputNumber）、删除单项
- 全选/单选：仅可勾选在售商品（status=1）
- 底部显示选中商品合计金额

**顶部角标**

- MainLayout 中显示购物车商品种类数量

#### 4.4.3 业务规则

| 规则 | 说明 |
|------|------|
| 自购限制 | user_id 与 goods.user_id 相同时拒绝加购 |
| 已售禁用 | 已售商品不可勾选结算 |
| 数量归零 | PUT quantity≤0 时自动删除购物车项 |

### 4.5 订单与交易模块

#### 4.5.1 功能描述

完成从结算、下单、付款、发货到确认收货的完整交易闭环，是本系统的核心业务模块。

#### 4.5.2 订单状态机

```
                    ┌──────────┐
                    │  待付款   │ status=0
                    └────┬─────┘
           取消订单      │ 付款
              ↓          ↓
        ┌──────────┐  ┌──────────┐
        │  已取消   │  │  待发货   │ status=1
        │ status=4 │  └────┬─────┘
        └──────────┘       │ 发货
                           ↓
                    ┌──────────┐
                    │  待收货   │ status=2
                    └────┬─────┘
                         │ 确认收货
                         ↓
                    ┌──────────┐
                    │  已完成   │ status=3
                    └──────────┘
```

#### 4.5.3 结算与下单

**两种入口**

1. **购物车结算**：选中若干购物车项 → `/checkout?cart_ids=1,2,3`
2. **立即购买**：商品详情 → `/checkout?goods_id=5`（先加购再下单）

**收货信息**

- 填写收货人、11 位手机号、详细地址
- 前端校验格式与长度

**拆单逻辑**

- 若购物车中商品来自不同卖家，后端按 `seller_id` 分组
- 每组生成独立订单，各自计算 `total_amount`
- 使用数据库事务（`beginTransaction`）保证原子性

**下单后**

- 清除对应购物车项
- 跳转第一个订单的详情页

#### 4.5.4 买家操作

| 状态 | 可用操作 |
|------|----------|
| 待付款 | 立即付款、取消订单 |
| 待收货 | 确认收货 |
| 其他 | 仅查看 |

付款为模拟支付，调用 `POST /api/orders/:id/pay` 后：
- 订单 status → 1（待发货）
- 关联商品 status → 2（已售）

#### 4.5.5 卖家操作

| 状态 | 可用操作 |
|------|----------|
| 待发货 | 填写物流单号/备注后发货 |
| 其他 | 仅查看 |

发货后订单 status → 2（待收货）。

#### 4.5.6 订单列表

- 路径 `/orders`，默认以买家视角查询
- Tab 筛选：全部、待付款、待发货、待收货、已完成
- 卡片展示订单号、状态、商品缩略、卖家、合计金额
- 点击卡片进入订单详情

### 4.6 即时消息模块

#### 4.6.1 功能描述

买家与卖家可在平台内直接沟通，协商价格、约定交易方式等，提升交易信任度。

#### 4.6.2 功能细节

**发起聊天**

- 商品详情页「联系卖家」→ 跳转 `/chat/:userId`
- 消息列表页点击会话 → 进入对应聊天

**消息列表**

- 调用 `GET /api/chat-list/:user_id`
- 展示所有有过聊天记录的用户及最后一条消息摘要

**聊天界面**

- 气泡式 UI，区分自己（右侧、主色）与对方（左侧、浅色）
- 发送消息调用 `POST /api/messages`
- 单条消息 ≤500 字

**实时推送**

- 前端 Socket.IO 连接后端，`join` 个人房间
- 新消息到达时触发 `new_message` 事件，自动刷新列表
- 消息区域自动滚动至底部

#### 4.6.3 通信架构

```
用户 A 发送消息 → POST /api/messages → 写入 messages 表
                              ↓
              io.to('user_B').emit('new_message')
                              ↓
              用户 B 前端监听 → 刷新消息列表
```

### 4.7 购物智能助手模块

#### 4.7.1 功能描述

集成 AI 对话能力，帮助用户发现商品、了解价格行情、熟悉购买流程，降低新用户使用门槛。

#### 4.7.2 功能细节

**对话界面**

- 路径 `/agent`
- 聊天气泡 + 快捷提问标签（「推荐一些好物」「有什么便宜的」「怎么购买」）
- 支持多轮对话，携带 history 上下文

**回复策略（双模式）**

1. **外部 LLM 模式**：配置环境变量 `SHOPPING_AGENT_API_URL` 与 `SHOPPING_AGENT_API_KEY` 后，将用户消息与在售商品数据一并发送至外部 API，返回 AI 生成回复
2. **内置规则模式**（默认）：基于关键词匹配的规则引擎
   - 「推荐/看看/买」→ 推荐前 3 件在售商品
   - 「价格/便宜/贵」→ 返回均价与最低价统计
   - 「流程/怎么买」→ 返回购买步骤说明
   - 其他 → 返回功能引导

**推荐卡片**

- 内置模式下，回复可附带推荐商品卡片
- 点击卡片跳转商品详情

**容错**

- 网络失败显示重试按钮
- 外部 API 调用失败自动降级至内置规则

### 4.9 管理员后台模块

#### 4.9.1 功能描述

#### 4.9.2 功能细节

**访问控制**

- 路由 `/admin/*` 需 `requiresAdmin` 守卫

**平台通知**（`/admin/notifications`）

## 五、核心业务流程

### 5.1 完整购物流程

```
浏览首页 → 搜索/点击商品 → 查看详情
    ↓
加入购物车（或立即购买）
    ↓
购物车勾选 → 去结算 → 填写收货信息 → 提交订单
    ↓
订单详情 → 付款（模拟）→ 等待卖家发货
    ↓
卖家发货 → 买家确认收货 → 交易完成
```

### 5.2 卖家发布与履约流程

```
登录 → 发布商品（描述+价格+图片）→ 商品上架
    ↓
（可选）买家联系聊天协商
    ↓
收到订单 → 买家付款 → 发货（填写物流信息）
    ↓
买家确认收货 → 交易完成
    ↓
（可选）手动标记已售
```

### 5.3 用户角色说明

本系统不区分固定「买家/卖家」角色，同一账号可同时进行购买与销售。界面根据当前用户与商品/订单的归属关系，动态展示对应操作按钮。

## 九、总结

「校园二手」平台基于 **Vue 3 + Express + MySQL** 技术栈，采用前后端分离架构，实现了校园社区二手交易的完整数字化流程。前端通过 Vue Router、Pinia、Ant Design Vue 构建响应式单页应用；后端通过 Express 提供 RESTful API，结合 Socket.IO 实现实时消息；MySQL 持久化用户、商品、订单与聊天数据。

系统涵盖用户认证、商品发布与浏览、购物车、订单履约、即时聊天、AI 购物助手等八大功能模块，满足校园师生「发现好物、发布闲置、安全交易、便捷沟通」的核心需求，具备较好的实用价值与扩展空间。
