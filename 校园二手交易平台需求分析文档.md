# 校园二手交易平台需求分析文档

---

## 一、需求背景与目标

### 1.1 业务背景描述

随着高校学生消费观念的转变和环保意识的增强，校园内二手物品交易需求日益增长。当前校园二手交易主要依赖微信群、QQ群、贴吧等渠道，存在信息分散、交易不安全、沟通效率低等问题。建立一个专门的校园二手交易平台，可以：

- 集中化展示校园内二手商品信息
- 提供安全可靠的交易担保机制
- 提升交易匹配效率和用户体验
- 促进校园资源循环利用

### 1.2 核心目标与成功指标

| 目标维度 | 具体目标 | 成功指标（6个月） |
|---------|---------|------------------|
| 用户规模 | 覆盖目标校园用户 | 注册用户达到校园人数的30% |
| 交易规模 | 促进二手交易活跃度 | 月活跃商品数≥500件，月成交订单≥200单 |
| 用户满意度 | 提供良好交易体验 | 用户评分≥4.0/5.0，投诉率<5% |
| 平台安全 | 保障交易安全 | 交易纠纷率<3%，欺诈事件0容忍 |

### 1.3 解决的问题与价值主张

**核心痛点：**
- 信息分散：交易信息散落在各类社交群组，查找困难
- 信任缺失：买卖双方互不信任，交易风险高
- 沟通低效：重复询价、议价过程繁琐
- 无保障机制：出现纠纷缺乏仲裁和保障

**价值主张：**
> "让校园二手交易更简单、更安全、更高效"

## 二、目标用户与使用场景

### 2.1 主要用户角色定义

| 用户角色 | 角色描述 | 核心诉求 |
|---------|---------|---------|
| **学生卖家** | 需要出售闲置物品的学生 | 快速变现、操作简单、交易安全 |
| **学生买家** | 需要购买二手物品的学生 | 物美价廉、品质有保障、交易便捷 |
| **教职工卖家** | 需要出售闲置物品的教职工 | 处理闲置、找到合适买家 |
| **平台运营方** | 负责平台管理和运营的人员 | 维护秩序、处理纠纷、促进活跃 |

### 2.2 用户痛点与当前解决方案

| 用户角色 | 当前痛点 | 现有解决方案缺陷 |
|---------|---------|----------------|
| 学生卖家 | 物品难以快速出手；定价没参考；担心被骗 | 微信群消息被淹没；定价随意；无保障机制 |
| 学生买家 | 找不到需要的物品；担心质量问题；价格不透明 | 信息分散难查找；无法验货；议价成本高 |
| 教职工 | 与学生交易渠道有限；物品类型不匹配 | 社交圈层差异；需求对接困难 |

### 2.3 典型使用场景描述

**场景1：毕业季大清理**
> 大四学生小李即将毕业，有大量教材、生活用品需要处理。他通过平台快速发布商品，设置合理价格参考，一周内成功售出80%的闲置物品。

**场景2：新学期装备采购**
> 大一新生小王想购买二手自行车和教材，通过平台搜索筛选，找到附近学长发布的商品，在线沟通后线下验货交易，节省了50%的开支。

**场景3：日常闲置流转**
> 研究生小张买了新电脑，想出售旧笔记本。通过平台发布，获得系统估价参考，使用担保交易功能，3天内安全完成交易。

## 四、用户故事

### 4.1 P0核心功能用户故事

#### US-001：校园身份认证

**用户故事：**
> As a 平台运营方
> I want 用户通过学工号/教职工号完成校园身份认证
> So that 确保平台用户的真实性和安全性

**验收标准：**
- [ ] 用户可输入学工号/教职工号进行身份验证
- [ ] 系统对接校园身份认证接口或支持人工审核
- [ ] 认证通过后方可使用交易功能
- [ ] 认证信息仅用于身份核验，保护用户隐私

---

#### US-002：发布二手商品

**用户故事：**
> As a 学生卖家
> I want 快速发布我的闲置物品信息
> So that 让潜在买家看到并购买我的商品

**验收标准：**
- [ ] 支持上传1-9张商品图片，单张不超过5MB
- [ ] 必填字段：标题、分类、价格、商品描述
- [ ] 选填字段：原价、新旧程度、取货方式
- [ ] 发布成功后商品进入审核状态，审核通过后展示
- [ ] 支持保存草稿功能

---

#### US-003：担保交易购买

**用户故事：**
> As a 学生买家
> I want 使用担保交易购买商品
> So that 我的资金安全有保障，避免被骗

**验收标准：**
- [ ] 买家下单后，资金进入平台冻结账户
- [ ] 卖家收到订单通知，准备发货/约定取货
- [ ] 买家确认收货后，资金自动释放给卖家
- [ ] 支持7天未确认自动确认收货机制
- [ ] 交易过程中可申请退款/退货，进入仲裁流程

---

#### US-004：商品搜索与筛选

**用户故事：**
> As a 学生买家
> I want 通过关键词和条件筛选找到需要的商品
> So that 快速定位目标商品，提高购物效率

**验收标准：**
- [ ] 支持关键词搜索商品标题和描述
- [ ] 支持按分类、价格区间、新旧程度筛选
- [ ] 支持按发布时间、价格排序
- [ ] 搜索结果支持分页展示，每页20条
- [ ] 无结果时显示推荐商品或搜索建议

---

#### US-005：买卖沟通

**用户故事：**
> As a 买卖双方
> I want 通过平台内置消息系统沟通
> So that 协商交易细节，保留沟通记录

**验收标准：**
- [ ] 买家可对感兴趣的商品发起咨询
- [ ] 支持发送文字消息和图片
- [ ] 消息实时推送，未读消息有提醒
- [ ] 聊天记录与订单关联，便于纠纷处理
- [ ] 禁止发送联系方式（手机号、微信等），引导站内交易

---

### 4.2 P1重要功能用户故事

#### US-006：交易评价

**用户故事：**
> As a 交易双方
> I want 在交易完成后互相评价
> So that 建立信用体系，帮助其他用户判断

**验收标准：**
- [ ] 交易完成后7天内可进行评价
- [ ] 评价维度：描述相符、交易速度、服务态度
- [ ] 支持文字评价和星级评分
- [ ] 评价内容公开显示，不可修改
- [ ] 累计评价形成用户信用等级

---

#### US-007：商品收藏与降价提醒

**用户故事：**
> As a 学生买家
> I want 收藏感兴趣的商品并接收降价提醒
> So that 在价格合适时及时购买

**验收标准：**
- [ ] 用户可收藏/取消收藏商品
- [ ] 收藏商品降价时推送通知
- [ ] 收藏列表支持分类查看和管理
- [ ] 商品下架时自动从收藏列表移除并通知

## 五、非功能需求

### 5.1 性能要求

| 指标项 | 要求 | 说明 |
|-------|------|------|
| 页面加载时间 | ≤2秒 | 首屏加载时间，3G网络环境下 |
| 接口响应时间 | ≤500ms | 95%的API请求响应时间 |
| 并发用户数 | ≥1000 | 系统可同时支持1000人在线 |
| 图片加载优化 | 懒加载+压缩 | 商品图片按需加载，压缩后≤100KB |
| 系统可用性 | ≥99.5% | 月度可用性指标，排除计划维护时间 |

### 5.2 兼容性要求

| 平台 | 支持范围 |
|------|---------|
| 移动端 | iOS 12+、Android 8+ |
| 浏览器 | Chrome 90+、Safari 14+、Edge 90+ |
| 屏幕适配 | 支持320px-414px宽度手机屏幕 |
| 小程序 | 微信、支付宝（P2阶段） |

### 5.3 安全要求

| 安全维度 | 具体要求 |
|---------|---------|
| 数据保护 | 用户敏感信息加密存储（手机号、身份证号） |
| 传输安全 | 全站HTTPS，TLS 1.2+ |
| 支付安全 | 对接合规支付渠道，不存储用户支付敏感信息 |
| 权限控制 | 基于角色的访问控制（RBAC），最小权限原则 |
| 内容安全 | 商品图片和描述自动审核+人工复审 |
| 防刷机制 | 接口限流、验证码防机器注册、防重复提交 |

### 5.4 体验要求

| 体验维度 | 具体要求 |
|---------|---------|
| 易用性 | 核心功能3步内完成，新用户无需教程即可上手 |
| 可访问性 | 支持字体大小调整，重要信息不仅依赖颜色传达 |
| 反馈机制 | 操作后即时反馈（加载状态、成功/失败提示） |
| 容错性 | 网络异常时友好提示，支持重试 |
| 一致性 | 统一的设计规范和交互模式 |

## 六、约束与边界条件

### 6.1 技术约束

| 约束项 | 说明 |
|-------|------|
| 校园身份认证 | 需对接学校统一身份认证系统或采用人工审核方案 |
| 支付渠道 | 需申请微信支付/支付宝商户资质，学生团队可能受限 |
| 服务器部署 | 建议云服务器部署，预算考虑共享型配置起步 |
| 存储成本 | 图片存储需控制成本，考虑CDN+对象存储方案 |
| 消息推送 | 依赖第三方推送服务，需申请相关权限 |

### 6.2 业务约束

| 约束项 | 说明 |
|-------|------|
| 合规要求 | 需遵守《电子商务法》，明确平台责任边界 |
| 禁售商品 | 需明确禁止交易的商品类型（食品、药品、虚拟商品等） |
| 交易限额 | 单笔/单日交易金额限制，防范风险 |
| 地域限制 | 初期仅限本校或合作校园，不开放社会用户 |

### 6.3 资源约束

| 约束项 | 说明 |
|-------|------|
| 开发周期 | MVP版本需在1-2个月内上线 |
| 团队规模 | 建议配置：2-3名开发 + 1名产品 + 1名运营 |
| 预算限制 | 初期以最小成本验证模式，控制服务器和第三方服务费用 |
| 第三方依赖 | 支付、短信、推送等依赖第三方服务稳定性 |

### 6.4 边界条件

**平台边界：**
- 平台仅提供信息撮合和资金担保，不介入商品质量检验（P2阶段可提供）
- 线下交付环节由买卖双方自行约定，平台提供建议但不强制
- 不承诺商品一定能售出，平台提供曝光但不保证成交

**功能边界（MVP）：**
- 暂不支持跨校交易（P1阶段考虑）
- 暂不支持物流快递，仅限线下自取或面交
- 暂不支持分期付款、信用卡支付等复杂支付方式
- 暂不支持虚拟商品交易
